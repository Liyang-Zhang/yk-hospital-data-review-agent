# 多线程开发规划

## 使用方式

本文件用于把后续工作拆成多个短线程，降低单线程上下文膨胀和 token 消耗。

每个新线程建议只聚焦一个主题，并在开头引用：

- [当前产品边界速记.md](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/当前产品边界速记.md)
- [执行级技术架构方案.md](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/执行级技术架构方案.md)

这样通常不需要再带整段长背景。

## Session 7：LLM 语义路由架构演进（待启动）

### 目标

在不破坏当前 Docker 部署版本和无 LLM Key 可运行能力的前提下，把当前规则优先路由逐步演进为 `Catalog-driven LLM Router + 固定统计工具执行 + AnswerabilityPolicy 安全闸门`。

### 重点输入

- [LLM语义路由架构演进方案-20260708.md](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/LLM语义路由架构演进方案-20260708.md)
- [执行级技术架构方案.md](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/执行级技术架构方案.md)
- [当前产品边界速记.md](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/当前产品边界速记.md)
- `app/yk_review_agent/services/intent_parser.py`
- `app/yk_review_agent/services/function_resolver.py`
- `app/yk_review_agent/services/metric_catalog.py`
- `app/yk_review_agent/services/answerability_policy.py`

### 推荐实施顺序

1. 增加 router mode 配置：
   - `rule_first`
   - `llm_shadow`
   - `llm_first`
2. 扩展 `route_trace`：
   - 记录规则路由结果
   - 记录 LLM 路由结果
   - 记录最终采用结果
   - 记录 policy 决策
3. 新增 `LLMSemanticRouter`：
   - 从 `MetricCatalog` 生成指标说明
   - 输出结构化路由结果
   - 不直接查库、不生成 SQL、不计算数值
4. 先启用 `llm_shadow`：
   - 用户结果仍由当前规则链路决定
   - 后台记录规则与 LLM 的差异
5. 再推进低置信和多候选问题启用 LLM：
   - 规则高置信问题继续直接命中
   - 多候选、长句、弱命中问题交给 LLM Router
   - 所有结果仍必须经过 `AnswerabilityPolicy`

### 成功标准

- 不配置 `LLM_API_KEY` 时，当前核心问答仍可运行
- 配置 `LLM_API_KEY` 后，可在 `llm_shadow` 模式看到 LLM 路由结果
- LLM 不会绕过医院权限、产品范围、指标范围和非法组合校验
- 多候选或低置信问题不再被规则部分命中卡死
- 不引入 LLM 直接 SQL 或自由统计口径
