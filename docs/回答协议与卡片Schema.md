# 回答协议与卡片 Schema

## 目的

本文档定义医院客户数据回顾智能体 V0.1 的正式回答协议。

目标不是做一个可自由作图的 BI 工具，而是做一个受控的统计问答产品。因此回答输出必须稳定、可解释、可嵌入，并且由后端掌控业务语义。

## 设计原则

1. 自然语言结论永远存在，是主答案。
2. 表格是证据层，由后端统一模板生成。
3. 图表是受控表达，不是开放式作图能力。
4. 前端负责渲染，不负责决定业务展示逻辑。
5. 用户问“看什么”，系统决定“用什么形式最合适”。

## 回答层级

### 1. Summary

- 必有
- 一段自然语言统计结论
- 必须能在没有表格和图表的情况下单独成立

适用场景：

- 总体送检量
- 整倍体率结论
- 质控概览
- 当前不支持 / 需要澄清

### 2. Table

- 有分组、分层、趋势明细时优先提供
- 由后端统一输出列名、排序和单位
- 前端只做通用表格渲染

适用场景：

- 按月 / 按季度 / 按日期趋势
- 按年龄分层
- 结果结构分布
- 质控分布

### 3. Chart

- 仅当指标存在稳定默认图形，且数据点足够时提供
- 当前不支持用户自由指定图表类型
- 当前不支持把聊天主链路做成图片生成器

V0.1 规则：

- 图表由后端返回结构化 schema
- 前端使用固定图表组件渲染
- 当分类点少于 2 个时，不出图

这条规则的目的，是避免只存在一个时间点或一个分组时仍然硬塞一张“形式上有图、信息上没增益”的图表。

## 当前统一 Schema

### ChatResponse

- `assistant_text`
- `structured_answer`
- `result_cards`
- `follow_up_suggestions`
- `trace_id`

### ResultCard

当前只允许三种卡片：

- `summary`
- `table`
- `chart`

### Summary Card

- `type = summary`
- `title`
- `content`

### Table Card

- `type = table`
- `title`
- `table.title`
- `table.columns`
- `table.rows`

### Chart Card

- `type = chart`
- `title`
- `chart.title`
- `chart.chart_type`
- `chart.categories`
- `chart.series`

当前图表类型只允许：

- `bar`
- `line`
- `pie`

## 图表选择原则

图表是否出现，以及默认类型，不由模型临时决定，而由指标契约决定。

示例：

- `PGT-A 送检量`
  - 默认图表目标：趋势
  - 默认图表类型：`bar`
- `PGT-A 整倍体率`
  - 默认图表目标：趋势
  - 默认图表类型：`bar`
- `PGT-A 年龄分层整倍体率`
  - 默认图表目标：分组比较
  - 默认图表类型：`bar`
- `PGT-A 质控概览`
  - 默认图表目标：分布
  - 默认图表类型：`bar`
- `PGT-A 结果结构分布`
  - 默认图表目标：分布
  - 默认图表类型：`bar`

## 明确不做的事情

- 不支持“给我换成折线图/雷达图/双轴图”的自由作图控制
- 不支持任意字段拖拽分析
- 不支持后端直接生成 PNG 作为主回答路径
- 不支持让模型自由决定图表类型和视觉细节

## 前后端职责边界

### 后端负责

- 判断问题是否可回答
- 生成自然语言结论
- 生成表格 payload
- 生成图表 payload
- 决定图表是否应该出现

### 前端负责

- 渲染 summary / table / chart 卡片
- 渲染 clarify / refuse 状态
- 承载追问与会话交互

## V0.1 结论

V0.1 的回答协议应固定为：

- `summary` 作为主答案
- `table` 作为结构化证据
- `chart` 作为受控增强表达

产品边界上，这是一套“统计结果展示协议”，不是“智能作图协议”。
