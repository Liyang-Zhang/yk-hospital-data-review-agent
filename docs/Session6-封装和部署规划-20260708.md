## 3. 最小可部署单元

### 3.1 后端 API

入口：

```bash
uvicorn yk_review_agent.main:app --host 0.0.0.0 --port 18765
```

当前 API 前缀：

```text
/api/v1
```

必须保留的后端能力：

- 健康检查
- demo metadata
- session 创建和会话上下文
- chat 问答
- snapshot 元信息透出
- `answer / clarify / refuse` 受控输出

### 3.2 前端工作台

开发启动：

```bash
cd frontend
npm install
VITE_API_BASE=http://localhost:18765/api/v1 npm run dev
```

生产构建：

```bash
cd frontend
npm ci
VITE_API_BASE=/api/v1 npm run build
```

部署形态建议：

- 本地开发：Vite dev server + 后端 API。
- 服务器交付：本轮默认交付前后端双容器，前端容器托管静态页面并在容器内代理 `/api/v1` 到后端。
- 外层 Nginx、域名、HTTPS、鉴权网关和正式发布由 IT 负责。

### 3.3 Snapshot 数据库

当前推荐生产执行源：

```text
SNAPSHOT_BACKEND=sqlite
SNAPSHOT_DB_URL=sqlite+pysqlite:///./snapshot.db
```

部署默认前提：

- 部署时默认 `snapshot.db` 已经准备好。
- 建库、验库、数据更新属于独立的数据准备流程，不作为服务容器启动的默认步骤。
- 后续数据源最终会替换为实时 API 接口，因此服务封装应把数据访问层视为可替换依赖。

独立建库命令：

```bash
python -m yk_review_agent.tools.build_snapshot_db --product all
```

独立检查命令：

```bash
python -m yk_review_agent.tools.inspect_snapshot_db --product pgta
python -m yk_review_agent.tools.inspect_snapshot_db --product pgtsr
```

部署约束：

- `snapshot.db` 属于数据产物，不应提交到 Git。
- 服务器上应把 `snapshot.db` 放到明确的数据目录，例如 `/opt/yk-review-agent/data/snapshot.db`。
- `.env` 中使用绝对路径，避免进程工作目录变化导致读取错误。
- 原始 Excel 快照可以不随运行服务常驻，但必须保留一份可追溯备份，用于重建数据库。
- Docker 部署时通过 volume 挂载 `snapshot.db`，不要把业务数据固化进镜像。

### 3.4 Docker 服务镜像

首选封装方式：

```bash
docker build -t yk-review-agent-backend:session6 .
docker build --build-arg VITE_API_BASE=/api/v1 -t yk-review-agent-frontend:session6 frontend
```

本地 compose 运行示例：

```bash
mkdir -p deploy/data
cp snapshot.db deploy/data/snapshot.db
cp deploy/backend.server.env.example deploy/backend.server.env
docker compose -f deploy/docker-compose.server.yml up -d
```

中国大陆服务器交付方式：

```bash
docker save yk-review-agent-backend:session6 -o yk-review-agent-backend-session6.tar
docker save yk-review-agent-frontend:session6 -o yk-review-agent-frontend-session6.tar
```

服务器导入：

```bash
docker load -i yk-review-agent-backend-session6.tar
docker load -i yk-review-agent-frontend-session6.tar
```

服务器启动时只需要确认：

- 两个镜像已 `docker load`。
- `snapshot.db` 已放到服务器约定路径。
- `backend.server.env` 或等价环境变量已配置。
- 宿主机端口已映射，例如前端 `18088 -> 80`、后端 `18765 -> 8000`。

## 5. 服务器部署方案

### 5.1 推荐第一版：Docker 离线部署

这是当前首选交付形态，适合中国大陆服务器环境：本地完成镜像构建和 `docker save`，再把 tar 包传到服务器 `docker load`。

组成：

- 后端：`yk-review-agent-backend` FastAPI 容器。
- 前端：`yk-review-agent-frontend` Nginx 静态前端容器。
- 端口：宿主机 `18765` 映射到后端容器 `8000`，宿主机 `18088` 映射到前端容器 `80`。
- 数据：服务器已有 `snapshot.db`，通过只读 volume 挂载进后端容器。
- 配置：服务器提供 `backend.server.env` 或等价环境变量。
- 代理发布：由 IT 基于端口信息自行配置外层 Nginx、网关、域名和 HTTPS。

优点：

- 服务器无需安装 Python/Node 依赖。
- 适合无法直接拉取外网镜像的环境。
- 镜像版本可以用 tar 包归档和回滚。
- 符合当前 SQLite snapshot mode。
- 后续替换实时 API 时，服务镜像和数据依赖边界更清楚。

风险：

- `snapshot.db` 更新需要明确替换和重启流程。
- SQLite 当前仍是过渡数据源，不应设计成长期多实例共享写入。
- 前端镜像默认使用相对路径 `/api/v1`，如果 IT 改变代理前缀，需要同步重建前端镜像。

### 5.2 本地打包与服务器导入

本地构建：

```bash
docker build -t yk-review-agent-backend:session6 .
docker build --build-arg VITE_API_BASE=/api/v1 -t yk-review-agent-frontend:session6 frontend
docker save yk-review-agent-backend:session6 -o yk-review-agent-backend-session6.tar
docker save yk-review-agent-frontend:session6 -o yk-review-agent-frontend-session6.tar
```

服务器导入：

```bash
docker load -i yk-review-agent-backend-session6.tar
docker load -i yk-review-agent-frontend-session6.tar
```

服务器 compose 运行示例：

```bash
docker compose -f docker-compose.server.yml up -d
```

交付给 IT 的关键信息：

- 前端监听：宿主机 `18088`
- 后端监听：宿主机 `18765`
- API 前缀：`/api/v1`
- 健康检查：`GET http://<server-ip>:18765/api/v1/health`
- 前端健康检查：`GET http://<server-ip>:18088/healthz`
- 容器内端口：`8000`
- 数据挂载：`/opt/yk-review-agent/data/snapshot.db -> /app/snapshot.db`
- 发布代理、域名、HTTPS、前端静态资源托管由 IT 负责

### 5.3 建库隔离说明

部署服务默认不执行：

- Excel 读取
- snapshot 重建
- snapshot inspect
- 数据口径核对

这些放到独立数据准备流程中完成。服务启动时只验证运行所需的 `snapshot.db` 能被读取；等后续切换实时 API 时，也应保持同样原则：服务依赖已配置好的数据访问入口，而不是在启动时做数据生产。

### 5.4 参考项目经验

已参考两个既有部署项目：

- `/home/zhangly/repo/github/yk-hpo-extraction-web`
- `/home/zhangly/repo/github/DeepRare`

`yk-hpo-extraction-web` 的可复用经验：

- 前后端分离项目适合保留 `deploy/` 目录，单独放集群/HPC 部署文件。
- 集群部署文档应明确区分“内网 Docker 试用”和“Nginx/公网发布”两个阶段。
- `docker-compose.hpc.yml` 应写清楚镜像名、端口、env file、healthcheck。
- `backend.hpc.env.example` 应把数据库、外部服务地址、CORS、默认项目号等运行配置显式列出来。
- 中国大陆或内网服务器如果不能拉镜像，应保留 `docker save / docker load` 路径。
- 前端构建时的 API 地址是高风险点，必须明确 `VITE_API_BASE` 指向直接后端端口还是 IT 代理后的路径。

`DeepRare` 的可复用经验：

- 单服务项目可以用一个容器承载 FastAPI 和静态调试页，避免过早拆复杂服务。
- entrypoint 可以统一默认环境变量、运行目录、缓存目录和启动命令。
- 运行数据、配置文件、只读知识库/数据库应通过 volume 挂载，不固化进镜像。
- 容器内使用固定路径，例如 `/app/runtime`、`/app/config`、`/app/database`，比依赖启动目录更稳。
- 对需要访问宿主机服务的场景，可以明确使用 `host.docker.internal` 或实际内网地址。

对当前项目的落地建议：

- 新增 `deploy/` 目录，放 `docker-compose.server.yml`、`backend.server.env.example`、部署说明和交付清单。
- 前后端先按双容器部署，不接管外层 Nginx、域名、HTTPS 和正式发布。
- `snapshot.db` 固定只读挂载到 `/app/snapshot.db`。
- `backend.server.env.example` 默认写 `SNAPSHOT_DB_URL=sqlite+pysqlite:////app/snapshot.db`。
- Dockerfile 可先保持简单，但建议补 healthcheck 或在 compose 中补 healthcheck。
- 后续如果前端由公司系统嵌入，应把 API base、CORS origin、登录态/医院上下文传入方式作为部署交接项。

### 5.5 不建议当前使用

暂不建议第一版就上：

- 在线自动重建 snapshot。
- 正式数据库替换 SQLite。

这些都可以做，但会把 Session 6 的重点从“交付闭环”拖到“平台工程”。

## 7. 成功标准

本地封装完成标准：

- 本地可以基于已准备好的 `snapshot.db` 构建前后端镜像并启动双容器。

- 服务器可以 `docker load` 前后端两个镜像 tar 包。
- 容器可以通过挂载的 `snapshot.db` 启动。
- `http://<server-ip>:18765/api/v1/health` 可访问。
- `http://<server-ip>:18088/healthz` 可访问。
- 通过 API 或 IT 代理后的前端能完成一次 `PGT-A` 问答。
- 通过 API 或 IT 代理后的前端能完成一次 `PGT-SR` 问答。
- `snapshot_metadata` 能正确显示当前数据边界。
- 服务重启后仍能读取同一份 `snapshot.db`。
- 已向 IT 提供宿主机端口、API 前缀、健康检查和数据挂载说明。
