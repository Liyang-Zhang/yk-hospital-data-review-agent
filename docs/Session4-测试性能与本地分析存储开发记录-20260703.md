# Session 4：测试性能与本地分析存储开发记录

## 目标

本 Session 不再只处理“慢测试”和“夹具瘦身”，而是同时解决以下三类问题：

1. 降低本地回归和调试成本
2. 降低当前业务快照问答链路的加载与分析耗时
3. 为即将到来的 `PGT-SR / PGT-AH / PGT-M` 真实统计执行建立可复用的本地分析存储方案

约束：

- 不主动改变现有业务口径
- 不把本项目演进成开放式 SQL 问答
- 继续保持“受控指标函数 + 受控筛选组合”的产品边界

---

## 为什么现在就要做

当前仓库已经不是“只有 PGT-A” 的准备阶段，而是已经进入“多产品快照已到位、真实执行即将扩展”的阶段。

已存在的业务快照：

- [PGTA数据统计输出-2025年.xlsx](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/PGTA数据统计输出-2025年.xlsx)
- [PGTSR数据统计输出-2025年.xlsx](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/PGTSR数据统计输出-2025年.xlsx)
- [PGTAH数据统计输出-2025年-上传微盘.xlsx](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/PGTAH数据统计输出-2025年-上传微盘.xlsx)
- [2025-PGTM类全年输出.xlsx](/home/zhangly/repo/github/yk-hospital-data-review-agent/docs/2025-PGTM类全年输出.xlsx)

当前代码状态：

- `PGT-A` 已有真实统计执行，但执行层仍依赖 Excel 读入后的 Python 内存过滤和聚合
- `PGT-AH / PGT-SR / PGT-M` 已接入快照元信息，但尚未进入真实执行链路
- 测试侧已经能看到大快照和函数级 cache clear 带来的额外成本

因此，这件事不应再被视为“以后再做的性能优化”，而应被视为后续多产品开发的前置基础设施。

---

## 当前实现问题

### 1. Excel 仍然承担执行主数据源角色

当前 `PGT-A` 主链路会先把 Excel 读成内存 records，再通过 Python 列表做过滤和聚合。

这条路径在单产品、样本量较小时还能接受，但在以下场景会变得越来越不稳定：

- 更多产品同时接入
- 更大的年度快照
- 更多维度组合
- 更频繁本地回归
- 更多 oracle 和 agent 测试同时覆盖

### 2. 当前查询是“内存全表扫描 + 重复派生”

目前可以确认的重复计算点包括：

- `eligible_records`
- `hospitals`
- `stat_month_range`
- `snapshot metadata`
- `data_readiness` 与 `query execution` 的重复过滤

即使 Excel 不是每次请求都重新读，后续分析阶段仍然在重复做 Python 侧全量扫描。

### 3. 测试夹具反复切换快照并清缓存

当前 oracle 测试会切换到更大的快照文件，并在测试过程中反复清理 dataset / snapshot registry cache。

这会带来两个问题：

- 回归耗时拉长
- 性能波动更大，不利于稳定评估

---

## 方案结论

本 Session 采用两层方案，但第二层不是“以后再考虑”，而是本轮就要启动设计并落第一版。

### 第一层：当前链路瘦身

目标：

- 降低测试回归耗时
- 降低当前 `PGT-A` 快照问答链路的重复计算

措施：

1. 缩减大快照测试依赖和函数级切换次数
2. 找出最慢的 agent / oracle 用例
3. 预计算常用派生数据
4. 减少一次请求中的重复过滤
5. 评估并补充过滤结果缓存

### 第二层：本地分析存储

目标：

- 让 Excel 退出执行主存储角色
- 建立按产品拆分的本地执行层
- 为 `PGT-SR / PGT-AH / PGT-M` 真实统计执行提供可替换的过渡数据源

结论：

- 本轮优先采用 `SQLite`
- Excel 保留为导入源和口径校验源
- 查询执行优先转到本地物化后的 SQLite 数据
- 当前不做跨产品表整合，不做统一事实表，不做表间关联

---

## 为什么选 SQLite

当前阶段更适合 SQLite，而不是继续扩 `openpyxl + Python list`，也不是立刻上更重的分析栈。

选择 SQLite 的原因：

1. 工程成本低
   - Python 生态接入轻
   - 项目中已存在数据库配置

2. 能满足当前查询形态
   - 按产品、医院、时间、年龄、结果标签筛选
   - 做计数、分组、汇总

3. 更适合作为“本地物化缓存”
   - 能承接 Excel 导入
   - 能建立索引
   - 能稳定支撑测试与本地开发

4. 对当前产品边界更合适
   - 继续保持“受控函数执行”
   - 不把系统演进成自由 SQL 问答

当前不建议：

- 继续把多产品执行建立在 Excel 内存扫描上
- 过早追求统一事实表、共享字段抽象或通用 BI 架构
- 为这批过渡快照设计长期稳定的数据模型

## 本地分析存储的设计方向

### 核心原则

1. `Excel` 是导入源，不是执行源
2. `SQLite` 是本地分析存储，不直接暴露给用户
3. 执行层仍然是受控指标函数，不开放自由查询
4. 按产品保留平表，不主动做跨产品整合
5. 当前快照库是过渡执行层，不提前模拟未来正式 API 的统一模型

### 建议的数据层次

建议至少分为两层：

1. 导入批次元数据
2. 分产品平表

当前建议直接按快照源 sheet 建 4 张主表：

- `pgta_snapshot_2025_data`
- `pgtsr_snapshot_2025_data`
- `pgtah_snapshot_2025_source`
- `pgtm_snapshot_raw_data`

对应来源分别是：

- `PGTA`：`2025年-数据`
- `PGTSR`：`2025年-数据`
- `PGTAH`：`2025源数据`
- `PGTM`：`原始数据`

设计原则：

- 每张表尽量贴近原 Excel 列结构
- 只补最小技术字段，不提前抽公共业务字段
- 产品之间不建立关联
- 跨产品能力仍停留在路由和 capability 层，不下沉到存储整合

### 索引建议

第一版至少需要：

- `hospital_name`
- `report_review_time`
- `month_bucket`
- `order_id`
- `sample_id`

如果按实际查询再补，可考虑联合索引：

- `(hospital_name, month_bucket)`
- `(hospital_name, report_review_time)`

## 分阶段实施建议

### Phase 1：测量与瘦身

目标：

- 建立性能基线
- 找出最慢回归子集
- 先压掉明显重复开销

建议动作：

1. 固定常用性能观测命令
2. 记录最慢 agent / oracle 用例
3. 调整 oracle 测试夹具作用域
4. 减少不必要的 cache clear
5. 预计算 `eligible_records / hospitals / stat_month_range`

### Phase 2：导入器与 SQLite 底座

目标：

- 打通 `Excel -> SQLite`
- 形成可重复执行的本地物化流程

建议动作：

1. 新增 snapshot importer
2. 为 `PGT-A / PGT-SR / PGT-AH / PGT-M` 建立导入配置
3. 每个产品按源数据 sheet 导入一张平表
4. 只补最小导入元数据字段
5. 建立基础索引

### Phase 3：PGT-A 查询迁移

目标：

- 让现有 `PGT-A` 真实执行先迁到 SQLite
- 用已有 oracle 和 agent 回归验证口径不变

建议动作：

1. 查询服务优先读 SQLite
2. 当前 `PGT-A` 指标函数逐步迁移
3. 对照现有 oracle 结果回归
4. 保留 fallback 方案，避免迁移初期阻塞开发

### Phase 4：多产品扩展复用

目标：

- 在统一底座上逐步接入 `PGT-SR / PGT-AH / PGT-M`

建议动作：

1. 先补产品数据契约
2. 再补指标函数
3. 最后补路由、answerability、oracle 回归

说明：

- 这里的“底座”指按产品拆分的 SQLite 执行层
- 不是统一事实表，也不是跨产品共享 schema
