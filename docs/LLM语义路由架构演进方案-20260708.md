# LLM 语义路由架构演进方案

## 1. 背景

当前项目已经从单一 `PGT-A` 统计助手，推进到支持 `PGT-A / PGT-SR` 的受控医院 PGT 数据回顾智能体。

当前主链路能稳定回答一批核心问题，但意图识别仍以规则优先：

```text
用户问题
  -> 规则解析
  -> FunctionResolver 关键词匹配与消歧
  -> 可选 LLM fallback
  -> AnswerabilityPolicy
  -> 固定统计函数
  -> 报告渲染
```

这种设计适合早期快速收敛产品边界，但随着后续指标、维度、产品线和用户问法增加，单纯扩展关键词和手写消歧会变得难维护。

核心风险：

- 指标越多，`FunctionResolver` 中的关键词冲突越多。
- 用户自然语言问法会远多于人工维护的 `hard_terms / soft_terms`。
- 当前 LLM 只是 fallback，规则一旦部分命中，LLM 可能无法及时介入。
- 如果直接让 LLM 生成 SQL 或计算结果，又会带来医疗数据统计口径和安全风险。

因此，后续推荐演进为：

```text
Catalog-driven LLM Router
  + 固定统计工具执行
  + 确定性 AnswerabilityPolicy 安全闸门
```

## 2. 当前架构定位

当前架构不是纯 LLM Agent，也不是传统 Intent Classifier，而是：

```text
Rule-first Controlled Router
  + Optional LLM Fallback
  + Fixed Analysis Functions
```

也就是说，当前系统主要依靠规则和指标目录完成：

- 产品识别
- 指标识别
- 时间、年龄、医院等筛选条件抽取
- 单指标函数选择
- `answer / clarify / refuse` 判断

LLM 当前只在存在 `LLM_API_KEY` 且规则没有明确跳过时才会参与。即使不配置 `LLM_API_KEY`，系统也可以依靠规则链路和固定统计函数正常回答一批问题。

这解释了为什么当前后端“看起来不依赖 LLM”：

- 数值计算由本地 SQLite snapshot 和固定函数完成。
- 答案文案大多由 report renderer / 模板生成。
- LLM 主要承担可选意图解析，不承担主计算。

## 3. 推荐目标架构

后续推荐目标链路：

```text
QuestionNormalizer
  -> ContextBuilder
  -> LLM Semantic Router
  -> AnswerabilityPolicy
  -> Metric Executor / Data API Adapter
  -> Report Renderer
```

### 3.1 QuestionNormalizer

负责低风险、确定性的预处理：

- 医院简称归一
- 年份、月份、季度表达归一
- 产品别名归一
- 常见空格、符号、大小写归一
- 会话上下文拼接

这一层可以继续保持规则实现，因为它是确定性文本标准化，不承担业务决策。

### 3.2 ContextBuilder

把用户问题和当前会话上下文整理成 LLM Router 可理解的输入：

- 当前医院
- 可访问医院范围
- 当前产品范围
- 上一次指标
- 上一次时间范围
- 上一次 breakdown
- 当前可用指标目录摘要

这一层的目标是让 LLM 做语义判断时不需要猜上下文。

### 3.3 LLM Semantic Router

LLM 不直接回答、不直接查库、不直接写 SQL，只输出结构化路由结果：

```json
{
  "request_kind": "single_metric",
  "product_scope": "PGT-A",
  "metric_id": "pgta_euploid_rate",
  "time_range": "2025年",
  "filters": {
    "hospital_id": "sxsfy",
    "age_range": ">35"
  },
  "breakdown": "age",
  "time_grain": "overall",
  "confidence": 0.86,
  "candidate_metric_ids": ["pgta_euploid_rate"],
  "ambiguity_reason": ""
}
```

LLM Router 的职责是“听懂人话并选择工具”，不是执行业务逻辑。

### 3.4 AnswerabilityPolicy

`AnswerabilityPolicy` 必须继续作为确定性安全闸门，负责最终判断：

- 是否是医院 PGT 数据回顾问题
- 产品是否支持
- 指标是否支持
- 医院权限是否允许
- 时间、年龄、临床指征等 filter 是否可执行
- metric + breakdown + time grain 组合是否合法
- 数据源是否具备所需字段
- 是否应该回答、澄清或拒答

即使 LLM Router 给出 `metric_id`，也必须经过这一层校验。LLM 说能答不算，策略层说能答才执行。

### 3.5 Metric Executor / Data API Adapter

统计执行仍然使用固定函数或后续数据 API：

- 不允许 LLM 直接生成 SQL。
- 不允许 LLM 临时发明统计口径。
- 不允许 LLM 修改指标公式。
- 所有计算必须来自已登记的 metric executor。

当前 SQLite snapshot executor 后续可以平滑替换为实时 API adapter，但对上层 Router 和 Policy 的契约应保持稳定。

### 3.6 Report Renderer

报告生成建议继续模板优先：

- 数字、分母、分子、时间范围、筛选条件来自 executor。
- 文案解释可以模板生成。
- 后续如引入 LLM 润色，只允许基于结构化结果做表达优化，不能重新计算或改写数值。

## 4. Metric Catalog 的角色升级

后续不应继续把 `MetricCatalog` 只当作路由规则辅助表，而应升级为统一的 `Tool / Metric Contract`。

每个 metric 应显式定义：

- `metric_id`
- 业务名称
- 业务解释
- 支持产品
- 支持 filters
- 支持 breakdowns
- 支持 time grains
- 数据字段要求
- 禁止组合
- 示例问法
- 同义表达
- 消歧说明
- 执行器绑定
- 输出结构

LLM Router 的工具描述应从 `MetricCatalog` 自动生成，而不是手写一份和代码可能漂移的 prompt。

这样新增指标时，理想流程是：

1. 增加 metric contract。
2. 增加 executor。
3. 增加 policy 组合校验。
4. 增加路由与答案回归用例。

而不是在多个 if/else 里补关键词。

## 5. 路由模式演进

建议通过环境变量控制路由模式：

```text
ROUTER_MODE=rule_first
ROUTER_MODE=llm_shadow
ROUTER_MODE=llm_first
```

### 5.1 rule_first

当前模式：

```text
规则路由为主
LLM 只做 fallback
```

适合离线演示、无 LLM Key、早期稳定验收。

### 5.2 llm_shadow

过渡模式：

```text
规则路由仍作为最终结果
LLM Router 后台同时解析
route_trace 记录两者差异
```

该模式用于积累评估数据，不影响用户结果。

建议记录：

- `rule_metric_id`
- `rule_candidate_metric_ids`
- `llm_metric_id`
- `llm_candidate_metric_ids`
- `llm_confidence`
- `final_metric_id`
- `policy_decision`
- `router_disagreement`

### 5.3 llm_first

目标模式：

```text
LLM Router 优先解析
AnswerabilityPolicy 强制校验
规则路由作为兜底和高置信快捷路径
```

适合指标和问法明显增长后的生产形态。

## 6. LLM 介入策略

不建议简单设计成“LLM 永远优先”或“LLM 永远 fallback”。

更合理的策略：

- 明显域外问题：规则预检可直接拒答。
- 明显高置信短问法：规则可直接命中。
- 多候选、长句、组合分析、弱命中：优先交给 LLM Router。
- LLM 低置信或返回多个候选：进入 clarify。
- LLM 命中不支持产品或非法组合：进入 refuse。

关键变化是：当规则层出现“部分命中、多候选、低置信”时，应该允许 LLM 介入，而不是跳过 LLM。

## 7. 不推荐的方向

### 7.1 不推荐 LLM 直接生成 SQL

原因：

- 医疗统计口径需要稳定可审计。
- SQL 正确不代表业务口径正确。
- 容易绕过医院权限、产品范围和字段语义约束。
- 回归测试难度高。

### 7.2 不推荐继续无限扩展关键词路由

关键词路由仍可用于预处理和高置信快捷路径，但不应继续承担全部语义理解。

如果后续每新增一个指标都要补大量 `hard_terms / soft_terms / if`，说明架构已经进入维护瓶颈。

### 7.3 不推荐过早引入复杂 Planner

普通单指标问答不需要 Planner。

Planner 更适合后续复杂问题，例如：

- 分析 PGT-A 某指标下降原因
- 对比不同医院、不同年龄层、不同临床指征
- 生成一份完整客户回顾报告

在进入 Planner 前，单指标 metric contract 和 policy gate 必须先足够稳定。

## 8. 分阶段实施计划

### Phase 0：保持当前主链路稳定

目标：

- 继续保证 Docker 部署版本稳定可用。
- 保留无 LLM Key 也可运行的能力。
- 不影响当前 PGT-A / PGT-SR 已支持问题。

### Phase 1：补充 Router Trace 和置信度

目标：

- 给规则路由增加置信度和命中证据。
- 记录候选指标、命中词、消歧原因。
- 为 shadow mode 做观测基础。

建议新增字段：

- `router_mode`
- `rule_confidence`
- `rule_evidence_terms`
- `ambiguity_reason`
- `should_call_llm`

### Phase 2：实现 LLM Router Shadow Mode

目标：

- 新增 `LLMSemanticRouter` 服务。
- 从 `MetricCatalog` 生成 LLM 可见的指标说明。
- 规则结果仍作为最终结果。
- 在 `route_trace` 中记录 LLM 解析结果。

验收重点：

- LLM 不影响现有答案。
- 可收集规则和 LLM 的不一致样本。
- 可发现规则误命中、漏命中和用户长尾表达。

### Phase 3：低置信与多候选问题启用 LLM

目标：

- 当规则低置信、多候选、长句复杂问法时，调用 LLM Router。
- LLM 输出仍必须经过 `AnswerabilityPolicy`。
- 低置信或多义结果进入 `clarify`。

这一阶段能直接缓解“规则部分命中导致 LLM 无法及时介入”的问题。

### Phase 4：LLM-first Router

目标：

- LLM Router 成为主要语义解析入口。
- 规则路由退化为：
  - 无 LLM Key 的离线兜底
  - 高置信快捷路径
  - 明显拒答预检

此阶段适合指标明显扩张后推进。

### Phase 5：受控 Planner

目标：

- 支持多指标、多步骤分析。
- Planner 只能选择已登记工具。
- 每一步仍经过 policy 校验。

示例：

```text
用户：分析今年 PGT-A 整倍体率下降的原因

Planner:
1. 查询年度整倍体率趋势
2. 查询年龄结构变化
3. 查询送检量变化
4. 查询质控变化
5. 汇总结构化解释
```

## 9. 测试要求

架构演进必须配套测试，否则很容易引入看似聪明但不可控的行为。

建议测试分层：

- 路由回归：同一批问法在不同 router mode 下的最终决策。
- Shadow 对比：规则和 LLM 的 metric 差异记录。
- Policy 防线：LLM 误选不支持指标时仍应 refuse。
- 权限测试：LLM 不能绕过医院访问边界。
- 组合测试：非法 breakdown / time grain 仍应 clarify 或 refuse。
- 无 Key 测试：不配置 `LLM_API_KEY` 时系统仍可启动和回答核心问题。
- 非 SQL 测试：确保 LLM 输出不能进入自由 SQL 执行链路。

## 10. 推荐结论

本项目后续最稳妥的 Agent 架构不是“纯规则”，也不是“LLM 自由发挥”，而是：

```text
LLM 负责听懂人话
MetricCatalog 负责定义能力
AnswerabilityPolicy 负责刹车
Executor 负责算数
Report Renderer 负责表达
```

这条路线可以同时满足：

- 用户自然语言问法扩展
- 指标数量增长
- 医疗统计口径可控
- 数据权限可控
- 后续从 SQLite snapshot 切换到实时 API
- 无 LLM Key 时仍可保留最小可运行能力

后续真正实施时，应优先从 `llm_shadow` 开始，而不是直接切换到 `llm_first`。
