# 医院客户定期数据回顾分析智能体，执行级技术架构方案

## 1. 目标

本项目面向医院客户的定期数据回顾场景，当前 `V0.1` 聚焦 `PGT-A` 单产品，通过聊天方式完成受控的数据分析问答；后续再扩展到 `PGT-SR`、`PGT-M`。

目标不是构建通用大模型 Agent，而是构建一个可嵌入企业系统、可审计、可扩展、可切换数据源的受控分析助手。

第一版要求：

- 支持单轮提问和基于当前会话的追问
- 支持结构化结论、表格、图表卡片输出
- 支持受控指标模板，不开放自由 SQL
- 支持后续嵌入各类公司内部系统
- 交付形式适合 IT 接手部署与集成
- LLM 提供层默认兼容 `Qwen / DashScope OpenAI-compatible` 接口

## 2. 范围与非范围

### 2.1 本次建设范围

- Python 后端服务
- 嵌入式聊天前端
- 当轮与当前会话级上下文管理
- 受控指标语义层
- 数据查询执行层
- 结果渲染层
- Docker 化交付
- IT 集成文档与环境配置文档

### 2.2 不纳入第一版

- 长期用户记忆
- 开放式数据库问答
- 多 Agent 协作编排
- 实时流式数据计算
- 独立 BI 平台能力
- 复杂权限平台建设

## 3. 总体架构结论

推荐采用 `逻辑前后端分离，部署统一` 的架构。

- 后端独立提供 API 和智能体内核
- 前端独立构建为嵌入式聊天工作台
- 发布时可统一打成一个 Docker 镜像
- IT 可通过 `iframe`、反向代理挂载或静态资源托管接入宿主系统

当前实现状态：快照模式 + 规则优先路由 + 单指标执行；真实执行主链路仍聚焦 PGT-A。

这样既能保持工程边界清晰，也避免第一版引入双系统独立运维负担。

## 4. 技术选型

### 4.1 后端

- Web 框架：`FastAPI`
- 配置管理：`pydantic-settings`
- Agent 编排：`PydanticAI`
- 数据访问：`SQLAlchemy Core`
- HTTP 客户端：`httpx`
- 运行服务：`uvicorn`

选型理由：

- FastAPI 适合 API、SSE、健康检查、后续 WebSocket 扩展
- PydanticAI 更适合受控工具调用和结构化输出，不引入重型编排心智负担
- SQLAlchemy Core 足够覆盖只读查询和后续视图访问，不强迫提前建复杂 ORM

### 4.2 前端

- 前端框架：`React + TypeScript`
- 构建工具：`Vite`
- 样式策略：原生 CSS + CSS Variables
- 图表：先输出统一 `chart_spec`，前端后续接 `ECharts`

选型理由：

- React 便于做嵌入式聊天壳和组件化结果卡片
- Vite 适合轻量构建和后续输出嵌入包
- 保持样式层轻，不依赖重 UI 库，方便 IT 接管和定制

### 4.3 交付

- 容器：`Docker`
- 本地环境：`Micromamba`
- 环境配置：`.env`
- 文档：OpenAPI + Markdown 接入说明

## 5. 模块划分

### 5.1 后端模块

后端拆分为 6 个核心模块：

1. `API Gateway`
   - 对外暴露聊天、会话、健康检查接口
   - 接收宿主系统透传的身份和医院上下文

2. `Session Context`
   - 管理当前会话消息历史
   - 管理时间范围、医院范围、产品类型、最近一次分析结果引用

3. `Conversation Orchestrator`
   - 调用大模型做意图识别、参数归一、工具路由
   - 执行拒答策略和追问补槽

4. `Metric Catalog`
   - 维护受控指标定义、统计口径、模板描述
   - 明确哪些问题允许回答，哪些问题直接拒答

5. `Query Service`
   - 只读查询执行
   - 参数校验、查询超时、模板绑定、后续审计留痕

6. `Report Service`
   - 把分析结果转为统一响应结构
   - 输出文字、表格、图表规范

### 5.2 前端模块

前端拆分为 5 个核心模块：

1. `Host Bridge`
   - 读取宿主系统透传的用户和医院上下文
   - 管理嵌入模式

2. `Chat Workspace`
   - 聊天输入、消息列表、加载与错误状态

3. `Result Cards`
   - 指标摘要卡
   - 表格卡
   - 图表卡

4. `Session State`
   - 管理当前会话消息状态
   - 管理追问输入

5. `API Client`
   - 对接后端会话与聊天接口

## 7. 会话上下文模型

只管理当前会话和当前任务级上下文，不做长期记忆。

### 7.1 会话级状态

- 会话 ID
- 用户 ID
- 医院 ID
- 创建时间
- 最后活跃时间

### 7.2 当前任务上下文

- 当前分析主题，例如 `PGT-A 整倍体率`
- 当前时间范围，例如 `2025 全年`
- 当前产品范围，例如 `PGT-A`
- 当前筛选条件，例如年龄段、季度
- 最近一次结果摘要

### 7.3 消息历史

- 用户消息
- 助手回复
- 内部结构化解析结果

## 8. API 草案

### 8.1 健康检查

`GET /api/v1/health`

返回：

- 服务状态
- 版本号
- 当前模式

### 8.2 创建会话

`POST /api/v1/sessions`

入参：

- 宿主系统会话 ID，可选
- 用户 ID
- 医院 ID

返回：

- 内部会话 ID
- 初始上下文

### 8.3 获取会话

`GET /api/v1/sessions/{session_id}`

返回：

- 会话元信息
- 当前上下文摘要
- 消息条数

### 8.4 发送聊天消息

`POST /api/v1/chat`

入参：

- 会话 ID
- 用户消息
- 宿主上下文

返回：

- assistant_text
- structured_answer
- result_cards
- follow_up_suggestions
- trace_id

### 8.5 导出

`POST /api/v1/export`

第一版可占位，后续支持：

- HTML
- PDF

## 9. 数据流

```text
用户输入
  -> 前端发送 chat 请求
  -> 会话服务加载当前上下文
  -> 编排层识别意图和参数
  -> 指标目录匹配可回答问题
  -> 查询服务执行模板查询
  -> 报告服务生成结构化结果
  -> 前端渲染文本、表格、图表卡片
```

## 10. 嵌入方案

### 10.1 推荐方案

优先级顺序：

1. `iframe 嵌入`
2. `同域静态资源挂载`
3. `微前端方式集成`

第一版推荐 `iframe` 或 `同域静态资源挂载`。

理由：

- 隔离宿主样式污染
- 降低系统耦合
- 便于独立升级

### 10.2 宿主系统需透传的信息

- 用户 ID
- 医院 ID
- 登录态 token 或签名
- 宿主会话 ID，可选

## 14. 风险与应对

### 14.1 数据口径风险

风险：

- 数据库字段存在但口径不等于业务统计口径

应对：

- 指标目录独立维护
- 数据适配层与指标口径分离

### 14.2 数据库性能风险

风险：

- 查询范围大，响应慢

应对：

- 只读视图
- 查询超时
- 固定模板
- 后续缓存

### 14.3 模型不稳定风险

风险：

- 自由生成内容不稳定

应对：

- 只允许受控问题路由
- 结构化输出
- 不确定时直接拒答

## 15. 实施顺序

1. 完成架构骨架和目录结构
2. 接通健康检查、会话接口、聊天占位链路
3. 补齐指标目录和 mock 数据
4. 对接真实模型
5. 对接真实数据库
6. 联调前端嵌入与宿主系统透传

- 项目目录与模块边界

真实数据库查询模板、正式 Prompt、导出模板和业务指标口径，将在下一轮迭代中逐步补齐。

### 16.1 当前开发收敛范围

结合当前可用数据，第一版实现收敛为：

- 使用业务人工整合的多产品快照文件作为当前主数据源
- 真实执行范围仅支持 `PGT-A`
- `PGT-AH / PGT-SR / PGT-M` 先接入快照元信息层和能力说明层
- 仅支持送检量、整倍体率、年龄分层、质控情况、嵌合与异常结构等可由当前 PGT-A 快照直接计算的问题
