# 执行级技术架构方案

## 1. 当前定位

本项目当前不是通用数据库问答 Agent，而是 **医院 PGT 数据回顾的受控统计助手**。

当前真实执行边界：

- 数据源：业务人工审核快照
- 产品：`PGT-A / PGT-SR` 第一阶段统计
- 形态：聊天式问答 + 结构化结果卡片
- 执行：受控函数，不开放自由 SQL

## 2. 当前主链路

当前问答主链路固定为：

1. `Question Normalization`
2. `Rule-first Intent Parsing`
3. `Follow-up Rewrite`
4. `Answerability Policy`
5. `Function Resolution`
6. `Query Execution`
7. `Report Rendering`

设计原则：

- 单问题优先保证稳定命中和稳定执行
- 多主题问题优先 `clarify`
- 域外、跨产品、非法组合优先 `refuse`
- 年龄范围始终作为全局 filter，而不是指标内部特例

## 3. 当前可执行函数

当前 `PGT-A` 真实执行范围包括：

- `pgt_total_volume`
- `pgta_euploid_rate`
- `pgta_quality_overview`
- `pgta_mosaic_abnormal`
- `pgta_cycle_indicator_overview`
- `pgta_special_cnv_overview`

当前 `PGT-SR` 第一阶段真实执行范围包括：

- `pgtsr_total_volume`
- `pgtsr_euploid_rate`
- `pgtsr_quality_overview`
- `pgtsr_result_overview`
- `pgtsr_cycle_indicator_overview`
- `pgtsr_next_step_overview`

所有新增能力都必须在这套函数体系内演进，除非明确进入下一阶段能力扩展。

## 4. 当前统一筛选维度

所有执行函数都应尽量统一承接：

- `hospital`
- `time_range`
- `age_range`

说明：

- `年 / 月 / 季度`：优先按业务统计月份字段
- `按日`：按审核时间字段
- 年龄支持：
  - `>35岁`
  - `35-37岁`
  - `未填写年龄`
  - 以及按业务年龄桶分层
- `PGT-SR` 还支持女性年龄、配偶年龄、临床指征等第一阶段筛选和分层，具体以 `MetricCatalog` 为准

## 5. 结果输出协议

当前结果协议固定为：

- `assistant_text`
- `structured_answer`
- `result_cards`
- `route_trace`
- `snapshot_metadata`
- `data_readiness`

其中：

- `route_trace` 用于验证路由一致性
- `result_cards` 用于验证答案一致性
- `snapshot_metadata` 用于显式说明当前数据边界

## 7. 当前已完成对齐

当前已完成并进入 oracle 回归的主题：

- 送检量
- 整倍体率
- 年龄分层整倍体率（作为 `pgta_euploid_rate + breakdown=age`）
- 年龄筛选后的整倍体率
- 质控总览
- 周期结局
- 结果结构
- 意外发现率

当前未纳入强制标准答案回归或仍需继续核口径的主题：

- 特殊 CNV 细分值
- 扩增成功率
- 某些尚未最终锁口径的窄口径指标
- 更深层的 `PGT-SR` 口径、oracle 基线和统计解释表达

## 8. 当前非范围

当前明确不承诺：

- `PGT-AH / PGTM` 真实执行
- 全产品汇总执行
- 复杂多指标联合执行
- 临床精细解释、病因推断、高龄/流产/种植失败等超出当前指标目录的统计
- 长期记忆或开放式临床解释

## 9. 后续演进原则

后续演进应遵循：

1. 先补 `PGTA` 标准答案基线覆盖
2. 再补长尾口语与追问继承
3. 再补前端补参工作流
4. 最后才考虑更多产品或更复杂任务

不要反过来先扩产品、扩复杂任务，再回头补单问题稳定性。

## 10. LLM 语义路由演进方向

当前架构仍以规则路由为主，LLM 只作为可选 fallback。随着后续指标、维度、产品线和自然语言问法增加，`FunctionResolver` 中的关键词匹配和手写消歧会成为维护压力点。

后续推荐演进为：

```text
QuestionNormalizer
  -> ContextBuilder
  -> LLM Semantic Router
  -> AnswerabilityPolicy
  -> Metric Executor / Data API Adapter
  -> Report Renderer
```

核心原则：

- LLM 负责语义理解和受控路由，不直接生成 SQL。
- `MetricCatalog` 升级为统一的 `Tool / Metric Contract`。
- `AnswerabilityPolicy` 继续作为确定性安全闸门，最终决定 `answer / clarify / refuse`。
- 执行层继续使用固定统计函数，后续可替换为实时 Data API adapter。
- 报告层模板优先，LLM 如参与只能做表达润色，不能改写数值。

建议先通过 `llm_shadow` 模式记录规则路由和 LLM 路由差异，再逐步推进低置信/多候选问题启用 LLM，最后再考虑 `llm_first`。
