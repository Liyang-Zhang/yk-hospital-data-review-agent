# 执行级技术架构方案

## 1. 当前定位

本项目当前不是通用数据库问答 Agent，而是 **医院 PGT 数据回顾的受控统计助手**。

当前真实执行边界：

- 数据源：业务人工审核快照
- 产品：仅 `PGT-A`
- 形态：聊天式问答 + 结构化结果卡片
- 执行：受控函数，不开放自由 SQL

## 2. 当前主链路

当前问答主链路固定为：

1. `Question Normalization`
2. `Rule-first Intent Parsing`
3. `Follow-up Rewrite`
4. `Answerability Policy`
5. `Function Resolution`
6. `Query Execution`
7. `Report Rendering`

设计原则：

- 单问题优先保证稳定命中和稳定执行
- 多主题问题优先 `clarify`
- 域外、跨产品、非法组合优先 `refuse`
- 年龄范围始终作为全局 filter，而不是指标内部特例

## 3. 当前可执行函数

当前 `PGT-A` 单产品真实执行范围只允许以下函数：

- `pgt_total_volume`
- `pgta_euploid_rate`
- `pgta_quality_overview`
- `pgta_mosaic_abnormal`
- `pgta_cycle_indicator_overview`
- `pgta_special_cnv_overview`

所有新增能力都必须在这套函数体系内演进，除非明确进入下一阶段能力扩展。

## 4. 当前统一筛选维度

所有 `PGT-A` 执行函数都应统一承接：

- `hospital`
- `time_range`
- `age_range`

说明：

- `年 / 月 / 季度`：优先按业务统计月份字段
- `按日`：按审核时间字段
- 年龄支持：
  - `>35岁`
  - `35-37岁`
  - `未填写年龄`
  - 以及按业务年龄桶分层

## 5. 结果输出协议

当前结果协议固定为：

- `assistant_text`
- `structured_answer`
- `result_cards`
- `route_trace`
- `snapshot_metadata`
- `data_readiness`

其中：

- `route_trace` 用于验证路由一致性
- `result_cards` 用于验证答案一致性
- `snapshot_metadata` 用于显式说明当前数据边界

## 6. 当前验证体系

当前验证已拆成两层：

### 6.1 路由层

验证：

- 是否命中正确函数
- 是否生成正确 filters
- 是否正确进入 `answer / clarify / refuse`

验证：

- 函数输出是否与业务标准答案一致

## 7. 当前已完成对齐

当前已完成并进入 oracle 回归的主题：

- 送检量
- 整倍体率
- 年龄分层整倍体率（作为 `pgta_euploid_rate + breakdown=age`）
- 年龄筛选后的整倍体率
- 质控总览
- 周期结局
- 结果结构
- 意外发现率

当前未纳入强制标准答案回归的主题：

- 特殊 CNV 细分值
- 扩增成功率
- 某些尚未最终锁口径的窄口径指标

## 8. 当前非范围

当前明确不承诺：

- `PGT-AH / PGT-SR / PGTM` 真实执行
- 全产品汇总执行
- 复杂多指标联合执行
- 临床指征、高龄/流产/种植失败等统计
- 长期记忆或开放式临床解释

## 9. 后续演进原则

后续演进应遵循：

1. 先补 `PGTA` 标准答案基线覆盖
2. 再补长尾口语与追问继承
3. 再补前端补参工作流
4. 最后才考虑更多产品或更复杂任务

不要反过来先扩产品、扩复杂任务，再回头补单问题稳定性。
