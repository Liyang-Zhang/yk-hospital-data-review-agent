# 结构化业务统计请求设计

## 背景

医院客户的真实需求并不总是单一指标问答，很多时候更像一张“统计任务单”。

示例：

- 指定时间范围
- 指定多个产品：PGT-A / PGT-SR（含 MARECS）/ PGT-M
- 指定一组业务指标：周期数、胚胎数、扩增成功率、NA 率、整倍体率、异常率、嵌合率、意外发现率

这类需求的本质不是普通聊天问答，而是 `structured business analysis request`。

## 产品边界

这类请求未来应支持，但必须以受控工作流方式支持，而不是让模型自由生成完整报告。

当前边界：

- 当前 V0.2 只执行业务审核快照 `PGTA数据统计输出-2025年.xlsx` 的 `PGT-A` 单指标统计
- 当前不执行跨产品、多指标汇总型请求
- 当前可以识别这类请求，并返回结构化能力缺口说明

## 核心处理链路

```text
Natural language request
        ->
AnalysisRequestParser
        ->
CapabilityMatrixBuilder
        ->
AnalysisPlanGenerator
        ->
Atomic metric functions
        ->
Structured result assembler
```

## 当前新增对象

### StructuredBusinessRequest

用于描述一个业务统计任务单，当前包含：

- `request_kind`
- `requested_products`
- `requested_metrics`
- `includes_marecs`

### CapabilityMatrix

用于描述当前产品版本和当前数据源下，对这次任务单的可执行能力：

- 按产品分组
- 每个产品下按业务指标列出 `supported / unsupported`
- 每个状态都附带原因

## 当前策略

对于结构化业务统计请求：

1. 先识别产品集合
2. 再识别指标集合
3. 生成 capability matrix
4. 当前 V0.1 统一进入 `refuse`
5. 但拒答内容不再只是“暂不支持”，而是包含：
   - 已识别产品
   - 已识别指标
   - 当前版本主限制
   - 分产品、分指标的能力说明

## 为什么当前不执行

当前不执行这类请求，不是因为产品方向不支持，而是因为下面几个前提尚未具备：

- 缺少跨产品的稳定事实层
- 缺少一套业务级原子指标函数
- 缺少统一的产品口径层
- 缺少对“部分支持”结果的正式前端展示模板

## 后续演进建议

### Phase 1

把当前识别和能力矩阵保留为主干，不让这类请求继续落到单指标 router。

### Phase 2

引入业务级原子指标函数，例如：

- `count_cycles(product, time_range)`
- `count_embryos(product, time_range)`
- `calc_na_rate(product, time_range)`
- `calc_euploid_rate(product, time_range)`

### Phase 3

允许在“同时间范围、同医院上下文”下执行多指标任务，并返回：

- 产品分段结果
- 指标矩阵表格
- 明确缺失项

### Phase 4

支持部分执行：

- 已支持项正常返回
- 不支持项明确列缺口
- 不再是全有或全无

## 当前结论

这类需求属于未来产品需要支持的核心能力，但它的正确实现路径是：

- 先识别成“结构化业务统计请求”
- 再做能力矩阵
- 再做可执行计划

而不是把它当成一句普通聊天问题直接去拼回答。
